Pros and cons med Github

Github har vært en fin platform å bruke i denne oppgaven. Den gjør det mulig for oss å dele kode og progressjon med hverandre. Oppdateringer og endringer blir ryddig og oversiktlig lagret i historikken så man kan gå tilbake og se på hva slags endringer som har blitt gjort, i tilfelle noe skulle ha gått galt underveis. Dette har vært spesielt nyttig når flere jobber på samme prosjekt, da man enkelt kan holde oversikt over hvem som har gjort hva.

En annen fordel med GitHub er muligheten til å opprette issues og pull requests, som har gjort samarbeidet enklere. Issues har gitt oss en strukturert måte å følge opp problemer og oppgaver på, mens pull requests har gjort det mulig å gjennomgå og kommentere hverandres kode før den blir slått sammen. Videre er GitHub veldig integrerbart med andre verktøy, som automatiserte testverktøy og CI/CD-pipelines, noe som kan effektivisere arbeidsflyten.

Selv om oppsettet av programmet gikk fint, kan det være noe komplisert eller overveldende for en som ikke har vært borti det før. Grunnleggende kommandoer som pull, push og merge krever tid å lære og kan skape frustrasjon for nye brukere. Et annet savn er muligheten til å se hverandres endringer i sanntid, noe som kunne forenklet samarbeidet ytterligere.

Vi støtte også på konflikter mellom hvilken kode som skulle stå hvor, spesielt i starten, hvor koden ikke var så stor enda og det var lett at områdene man jobbet på overlappet. Selv om GitHub tilbyr verktøy for å håndtere slike konflikter, krever det tid å lære og kan forstyrre arbeidsflyten. For private prosjekter kan det også oppstå kostnader dersom man ønsker tilgang til visse avanserte funksjoner.
